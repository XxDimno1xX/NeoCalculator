## A. Schema (restated from Doc 3 §D — normative)

Fields per row: `id, category, input, input_form, mode, engine, variables, assumptions, expected_kind, expected_canonical, accepted_alternates, forbidden_outputs, oracle, timeout_ms, memory_budget, tags, notes, source`; plus `sample_range`/`sample_points` only on rows using O4/O5. `assumptions` must be `[]` (reserved, Doc 3 §D). Unknown fields are a load error; ids are immutable once merged.

## B. Status markers (encoded in `tags` — closed vocabulary)

| Tag | Meaning | Gating |
|---|---|---|
| `status:pass-now` | expected to pass against current source | gates once the runner exists |
| `status:xfail` | pins a known bug; expected to FAIL until its ticket closes (always with `known-bug` + ticket ref in `notes`) | reported, never gates; XPASS gates |
| `status:blocked-d1` / `-d2` / `-d3` | expectation depends on an unratified decision (`NUMOS_CAS_SEMANTICS_DECISION_RECORDS.md`); runs report-only | never gates until ratification |
| `giac-only` | only Giac computes this; always paired with `firmware-only` | hardware lane (CAS-15) |
| `native-cas-only` | exercises in-tree `cas::` specifically (not VPAM/RPN/Giac) | host runner |
| `display-only` | asserts rendering/mode behavior, not engine value | emulator/display vehicle |
| `property` | metamorphic/property row (O7); checks consistency, not a pinned value | host runner |

## C. `input` micro-grammar for `input_form: "symexpr"`

The CAS-02 runner predates the CAS-03 normalizer, so `symexpr` inputs use this minimal, fully specified grammar (runner-local recursive-descent, ~150 lines):
`VERB ":" EXPR` where `VERB ∈ {simplify, evaluate, diff, integrate, solve}`; `solve` takes `LHS "=" RHS`. `EXPR`: integers, rationals `p/q`, variables `x y`, `+ - * / ^`, parentheses, `pi`, `e`, functions `sin cos tan arcsin arccos arctan ln log log2 abs exp`, unary minus. `diff`/`integrate` are with respect to `x`; `solve` with respect to the single variable present. Implicit multiplication is NOT allowed in this grammar (always explicit `*`) — parser-ambiguity rows live in the RPN/keys vehicles instead. `expected_canonical` for symexpr rows is the runner's serialization of the simplified result tree (same grammar, canonical child order per `SymExpr.h:499-504`, minimal parentheses); for `keys` rows it is the `assert_result` canonical text (`src/hal/NativeHal.cpp:1543-1557`).

For `input_form: "keys"`, `input` holds the human math text; `scripts/corpus-to-numos.py` projects it to key sequences (vocabulary per existing scripts, e.g. `tests/emulator/scripts/calc_fraction_sum.numos`: `key 1`, `key FRAC`, `key +`, `key RIGHT`, `key ENTER`, `assert_result`). Rows whose keys projection is not yet expressible (e.g. Root-glyph degree entry) carry `"tags":[…,"host-first"]` and run host-side until the projector supports them.
